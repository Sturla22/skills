# Product Brief

## Storage

- Work ID: `scenario-traceability`
- File path: `docs/work/scenario-traceability/brief.md`

## Request summary

When working on a project, keep a living list of high-level usage scenarios in
plain human-readable language. Implement those scenarios as tests with explicit,
mechanically verifiable traceability between each scenario and the tests that
cover it.

## Problem / desired outcome

There is currently no convention in this starter repo for capturing usage
scenarios at a level that non-technical stakeholders can read and review, nor
any link between those scenarios and the automated tests that demonstrate them.
As a result:
- Important behavior lives only in test code, invisible to stakeholders
- Coverage gaps are discovered late
- Handoffs lose intent — a test exists but its connection to a stated user need
  is implicit at best

**Desired outcome:** Any project using this starter repo can maintain plain-English
usage scenarios with stable IDs, write tests that reference those IDs, and run a
script that confirms every scenario is covered by at least one test.

## Why this matters

Traceability from stated need → scenario → test is the minimum bar for credible
verification. Without it, "all tests pass" does not answer "did we build what
was intended?"

## Stakeholders / users

- **Firmware team developers** — write tests that reference scenario IDs
- **Product owners / requester** — author and maintain the plain-English
  scenarios; the primary audience for the scenario files
- **Reviewers / verifiers** — use the trace table and coverage script to confirm
  completeness
- **Downstream teams adopting the starter repo** — inherit the convention and
  tooling

## Stakeholder needs / system outcomes

1. Scenarios are readable by non-technical stakeholders without requiring code knowledge
2. Every scenario has a stable, unique ID that survives editing
3. Tests explicitly reference the scenario(s) they cover
4. A script can detect when a scenario has no test coverage and exit non-zero
5. A trace table gives a human-readable map from scenario → test locations
6. The convention works at both project level and per-feature (work-packet) level
7. The guidance is durable — part of the repo's documented workflow, not just chat

## Design criteria / key parameters

- **Scenario format**: plain English, not Gherkin — no Given/When/Then ceremony required
- **Scenario ID format**: `SC-NNN` (three-digit zero-padded integer) at both
  project and work-packet level; the file path provides scope context
- **Test reference format**: a short language-neutral comment or annotation,
  e.g. `// Covers: SC-001` or `# Covers: SC-001` — exact style is
  language-appropriate but the keyword `Covers: SC-` must be machine-parseable
- **Trace table**: a Markdown table (scenario ID, description, test file(s), status)
  maintained alongside the scenarios file; can be regenerated by the script
- **Coverage script**: language-agnostic — scans scenario files for `SC-NNN`
  IDs, scans test files for `Covers: SC-NNN` references, reports gaps, exits
  non-zero on any uncovered scenario; also flags orphaned test references
  (a test covers an ID that appears in no scenario file)
- **Script location**: `scripts/check-scenario-coverage` (shell or Python, to
  be decided by planner; Python preferred for robustness)

## In scope

- `templates/scenarios-template.md` — reusable template for a scenarios file
  (used at both project and work-packet level)
- `scripts/check-scenario-coverage` — the coverage verification script
- Convention documentation: where scenarios live, ID format, test reference
  syntax, trace table format
- Guidance wired into CLAUDE.md and/or AGENTS.md so the convention is a
  default practice
- Optionally a new skill or updates to existing `bdd` / `requirements-and-traceability`
  skills to point to this convention

## Out of scope

- CI integration (convention and script are ready to integrate; wiring into CI
  is a separate step per project)
- Gherkin/Cucumber-style scenario files
- Auto-generating tests from scenarios
- Test framework-specific adapters or plugins
- Generating scenarios from code
- Per-language test annotation standards (one cross-language comment convention
  is sufficient)

## Constraints

- The script must work on a clean shell with either Python 3.x or POSIX sh
  (no additional runtime dependencies beyond what ships with a standard dev box)
- The scenarios file format must be readable without tooling — plain Markdown
- Scenario IDs must be stable across edits (do not re-number when inserting)
- The convention must not conflict with existing `bdd` or
  `requirements-and-traceability` skill guidance — extend, not contradict

## System context / external interfaces

- Adds to the existing work-packet model: `docs/work/<work-id>/scenarios.md`
  alongside `brief.md`, `plan.md`, `status.md`
- Adds a project-level entry point: `docs/scenarios.md`
- Adds `scripts/` directory (may be new)
- Touches `templates/`, `CLAUDE.md`, `AGENTS.md`, and possibly one or two skills

## Acceptance criteria

**SC-001** — A scenarios file created from the template is readable plain English
with no tool required; each scenario has a unique `SC-NNN` ID and a one-sentence
description.

**SC-002** — A test file that includes the comment `Covers: SC-NNN` is
recognized by the script as covering that scenario.

**SC-003** — Running the coverage script against a project where one scenario
has no test coverage produces a non-zero exit code and names the uncovered
scenario(s).

**SC-004** — Running the coverage script against a project where all scenarios
are covered produces a zero exit code.

**SC-005** — Running the script against test files that reference an `SC-NNN`
ID not present in any scenarios file reports it as an orphaned reference.

**SC-006** — The script discovers scenarios files at both `docs/scenarios.md`
and `docs/work/*/scenarios.md` automatically.

**SC-007** — The template and guidance are sufficient that a developer can set
up the convention on a new project without reading the work packet — the
template and CLAUDE.md/skill guidance are self-contained.

**SC-008** — A trace table in the scenarios file correctly maps each SC-NNN to
the test files that cover it (human-maintained or script-generated).

## Measures of effectiveness / performance

- Script runs in < 5 s on a repo with 200 test files
- Template is ≤ 60 lines including comments and examples
- Guidance addition to CLAUDE.md is ≤ 3 bullet points

## Behavior rules / examples (BDD)

**Rule: Every scenario must have a unique stable ID**
- Example: `SC-001` assigned on creation; inserting a new scenario between
  SC-001 and SC-002 does not renumber existing IDs; new scenario gets the next
  available integer

**Rule: Tests opt in by reference, not by location**
- Example: a test file in `tests/unit/` that contains `// Covers: SC-003`
  counts as covering SC-003 regardless of file name or directory

**Rule: The script is the single source of truth for coverage status**
- Example: the trace table is informational; the script re-derives coverage
  status from the actual test files on every run

## Derived requirements / traceability notes

- Scenarios files must use a parseable pattern (`SC-[0-9]{3,}`) so the script
  does not need to understand Markdown structure beyond line scanning
- The script must accept a configurable test file glob (default: `tests/**/*`)
  to stay framework-agnostic
- The script must accept a configurable scenarios file glob (default:
  `docs/scenarios.md docs/work/*/scenarios.md`) to support both scopes
- Trace table format must be renderable in any Markdown viewer (standard GFM table)

## Public contract / compatibility impact

This work adds new files and conventions to the repo. It does not remove or
rename any existing documented artifact. Existing skills and templates continue
to work unchanged; this extends them.

## Delivery class

**Workflow evolution** — adding a new reusable practice and tooling to the
starter repo itself. The scenarios template and coverage script are
non-productized tools; the convention guidance is part of the repo's documented
workflow (product development of the repo).

## TDD expectation

The coverage script is a non-productized tool. TDD is not required by default,
but explicit verification is required: the script must be exercised against
a fixture directory that exercises each acceptance criterion (SC-001 through
SC-008) before the work is closed. A lightweight test fixture (small synthetic
scenarios + test files) is sufficient.

## Validation intent / evidence

The primary stakeholder-fit question: "Is the convention simple enough that a
product owner would actually write the scenarios file, and a developer would
actually add the coverage comment?" Validation is answered by the requester
reviewing the template and sample scenarios before work is closed.

## SemVer / changelog expectation

**MINOR** — new templates, new script, new guidance. No existing documented
contract is broken. Changelog entry under `Added`.

## Assumptions

- The `scripts/` directory does not yet exist; creating it is in scope
- Python 3.x is available on any dev machine using this repo (preferred for
  the script over shell for robustness)
- Scenario IDs are project-global within one scenarios file; two files may
  independently both have `SC-001` and the file path disambiguates them
- The trace table is human-maintained by default; the script can optionally
  regenerate it but this is not a hard requirement for v1
- No existing skill needs to be removed; only extended or cross-referenced

## Open questions

1. **Script output format**: plain text report vs. Markdown vs. JSON? Plain
   text is assumed; planner to confirm.
2. **Trace table regeneration**: should the script offer a `--update-table`
   flag to write the trace table back to the scenarios file? Nice-to-have;
   planner to scope.
3. **ID format flexibility**: should the script accept `SC-1` and `SC-001` as
   the same ID, or enforce zero-padding? Enforce zero-padding for consistency
   is the default assumption.
4. **AGENTS.md vs CLAUDE.md vs new `.claude/rules/` file**: where should the
   convention guidance live? Planner to decide based on length and scope.
